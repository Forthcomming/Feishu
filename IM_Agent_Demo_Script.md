# 比赛演示脚本（CLI 融合架构版）

> 目标：在 6-8 分钟内清晰展示命题 Must-have，并体现 `Agent 主驾驶、GUI 辅助`。

## 1. 演示准备

## 1.1 环境准备
- 启动服务：
  - `agent-service`
  - `realtime-server`
  - `frontend`
- 准备两个端：
  - 桌面端页面（默认）
  - 移动端页面（`?device=mobile`）
- 准备飞书测试群：
  - 推荐 bot 自建私有群，避免外部群策略影响。

## 1.2 验收口径
- 必须体现：IM + Doc + PPT(或过渡演示材料) 三套件串联。
- 必须体现：同一任务在移动端和桌面端实时同步。
- 必须体现：由 Agent 编排执行，不是前端写死流程。

## 2. 演示主线（标准版）

## Step 1：IM 指令触发（场景 A）

### 操作
- 在桌面端输入：
  - “帮我把刚才讨论整理成需求文档，并给出汇报材料大纲，完成后回群通知。”

### 讲解台词
- “这里不是点按钮触发固定流程，而是自然语言进入 Agent。”
- “系统会把指令发送到 Orchestrator，先做意图识别和步骤规划。”

## Step 2：任务理解与规划可视化（场景 B）

### 操作
- 展示任务面板状态推进：
  - `detecting -> intent -> planning -> executing`

### 讲解台词
- “我们把任务拆成显式步骤，例如提取要点、创建文档、交付回传。”
- “每一步都有事件日志，失败可定位、可重试。”

## Step 3：文档生成与编辑（场景 C）

### 操作
- 展示生成的飞书文档链接或文档预览块。
- 在页面做一次轻量修改（例如补充一条需求点）。

### 讲解台词
- “文档由 Agent 驱动 `lark-cli docs` 创建，不是本地假数据。”
- “GUI 只负责展示和精修，核心执行仍由 Agent 控制。”

## Step 4：演示稿能力展示（场景 D）

### 操作（两种任选一种）
- 方案 1：展示 Slides 创建结果链接。
- 方案 2：展示当前项目的 PPT 预览能力，并说明 Phase 2 接入真实 Slides。

### 讲解台词
- “文档结构可直接映射成演示材料，减少重复搬运。”

## Step 5：多端实时一致（场景 E）

### 操作
- 在桌面端触发任务后，切到移动端展示相同任务状态和产物链接。

### 讲解台词
- “多端不是各自算状态，而是共同消费同一任务事件流。”
- “因此桌面和移动端看到的是同一份任务真相。”

## Step 6：总结与交付回 IM（场景 F）

### 操作
- 展示 Agent 自动回 IM 的消息（包含文档/演示链接）。

### 讲解台词
- “最后一步由 Agent 自动回传交付结果，形成从 IM 到交付的闭环。”

## 3. A-F 场景与演示动作映射

- A 意图入口：Step 1
- B 任务规划：Step 2
- C 文档生成：Step 3
- D 演示稿生成：Step 4
- E 多端一致：Step 5
- F 总结交付：Step 6

## 4. 失败兜底脚本（现场救场）

## 4.1 IM 权限失败
- 话术：
  - “当前租户策略限制了该身份发送，我们切换到 bot 自建群演示同流程，核心架构不变。”
- 预案：
  - 预置 bot 群 `chat_id`，切换 `--as bot`。

## 4.2 Slides 未就绪
- 话术：
  - “当前版本已完成文档闭环与多端同步，Slides 已按同一契约预留，在下一阶段接入。”
- 预案：
  - 展示 PPT 预览页和已定义的步骤计划。

## 4.3 网络波动
- 话术：
  - “任务事件存储可回放，页面刷新后仍能恢复任务状态。”
- 预案：
  - 打开 `GET /workflow/:taskId` 展示任务快照。

## 5. 评委高频问题答法（简版）

- “为什么用 CLI 不是直连 SDK？”
  - “CLI 让 IM/Doc/Slides 统一进一个 Tool Adapter，缩短集成周期并提升可观测性。”

- “怎么体现 Agent 是主驾驶？”
  - “状态机和步骤执行都在 Orchestrator，前端只展示事件和确认节点。”

- “多端一致怎么保证？”
  - “以服务端任务事件为唯一真相，两端订阅同一 task channel。”
